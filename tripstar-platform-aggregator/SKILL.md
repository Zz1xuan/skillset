# TripStar Platform Aggregator

在 Minis 中执行“多平台旅行信息检索 → 去重提纯 → 行程规划 → 可视化输出”的工作流。目标不是复刻某个前后端项目，而是复用其核心决策逻辑，稳定产出可执行旅行方案。

## 适用场景

当用户要：
- 做目的地旅行规划
- 搜索小红书/抖音等平台攻略
- 汇总多个平台的景点、美食、住宿、交通信息
- 生成 1~N 天行程
- 输出预算、预约提醒、避坑建议
- 导出 Markdown / HTML 旅行页

## 默认原则

- 默认优先“公开可访问 + 稳定”的信息源。
- 不把单一平台当唯一真相来源；至少交叉 2~3 类来源。
- 平台受限时自动降级到网页搜索、地图、官方旅游站、点评站。
- 对小红书、抖音类平台，只承诺“尽力提取公开可见内容”，不要承诺稳定全量抓取。
- 先输出结构化结论，再补充来源说明。

## 推荐信息源优先级

按任务选择，不要求每次全用：

1. 地图/POI
   - 高德地图
   - Google Maps
2. 攻略/真实体验
   - 小红书
   - 抖音
   - 马蜂窝
   - 大众点评
3. 预订/价格参考
   - 飞猪
   - 携程
   - Booking / Agoda（如适用）
4. 官方与公共信息
   - 景区官网
   - 文旅局/博物馆/公园官方页面
   - 天气服务
5. 搜索引擎补充
   - 使用 web-search skill 或浏览器搜索补足公开网页

## 输入信息

尽量从用户原话中提取；缺少关键字段时再补问。

核心字段：
- destination
- date_range 或 days
- travelers
- budget
- interests
- transport_preference
- hotel_preference
- pace_preference
- food_preference
- language

可选字段：
- must_visit
- avoid_list
- child_or_elderly_constraints
- accessibility_needs
- photo_spot_preference
- shopping_preference
- visa_or_cross_border_notes

## 工作流

### 1. 解析需求

把自然语言整理成结构化 brief：
- 目的地与范围
- 天数与日期
- 人群画像
- 预算等级
- 偏好与禁忌
- 输出语言

如果用户要求“搜某平台”，把该平台标记为优先来源，但保留降级方案。

### 2. 选择信息源组合

按目的选择来源：

- 查景点分布 / 顺路关系：高德、Google Maps
- 查真实体验 / 避坑 / 出片：小红书、抖音、马蜂窝
- 查吃喝：大众点评、高德、小红书
- 查住宿落点：飞猪、携程、Booking、Google Maps
- 查开放时间 / 预约：景区官网、官方公众号网页镜像、地图详情页
- 查天气：稳定天气源或官方天气服务

### 3. 搜索与提取

对每类来源提取：
- 名称
- 类型
- 区域/地址
- 推荐理由
- 典型玩法
- 预计时长
- 价格区间
- 预约要求
- 风险/避坑点
- 来源链接或平台标记

对内容平台特别提取：
- 高频被提到的景点
- 高频踩坑词
- 高频排队/预约提醒
- 高频美食店名
- 最佳时段/机位/路线建议

### 4. 去重与可信度分层

把信息分成：
- 官方事实：开放时间、预约、票价规则、地址
- 地图事实：位置、距离、路线、营业标签
- 用户经验：体验、踩坑、拍照、美食口味
- 模型推断：路线优化、节奏安排、预算估算

冲突时优先级：
1. 官方页面
2. 地图/POI 平台
3. 多个用户内容一致结论
4. 单条用户经验
5. 纯模型补全

### 5. 生成旅行方案

至少输出：
- 总体策略
- 每日 itinerary
- 景点顺序与理由
- 餐饮建议
- 住宿区域建议
- 交通建议
- 预算明细
- 预约提醒
- 避坑提示
- 备选方案（如下雨 / 人太多 / 预算超标）

### 6. 可视化输出

默认可输出：
- Markdown 攻略（默认）
- 单文件 HTML 页面（用户明确需要时）
- 结构化 JSON（仅在用户明确要求机器可读结果时）

如用户未指定格式，默认只输出 Markdown 结果，不额外生成 JSON。

如用户明确要“更像应用展示”，优先调用 `generative-ui-minis` skill 生成包含卡片、预算表、时间线、注意事项的 HTML 页面。

## 环境变量与登录态

可选增强模式：当用户希望提升小红书/抖音公开内容读取成功率时，可通过环境变量提供站点 cookie。

建议环境变量：
- `XHS_COOKIE`：小红书网页端登录后的完整 Cookie
- `DOUYIN_COOKIE`：抖音网页端登录后的完整 Cookie

使用原则：
- 仅在用户明确希望启用增强模式时使用。
- 仅用于请求对应站点的公开可见内容，不承诺可访问私有或受限数据。
- 若环境变量不存在，自动回退到“搜索引擎索引页 + 地图 + 点评 + 官方页”的默认路径。
- 不要在对话中输出 cookie 原文，不要打印环境变量值。

检查方式：
- 用 `[ -n "$XHS_COOKIE" ] && echo set || echo not set`
- 用 `[ -n "$DOUYIN_COOKIE" ] && echo set || echo not set`

若变量缺失，可提示用户创建：
- [Set XHS_COOKIE](minis://settings/environments?create_key=XHS_COOKIE&create_value=&create_note=%E5%B0%8F%E7%BA%A2%E4%B9%A6%E7%BD%91%E9%A1%B5%E7%99%BB%E5%BD%95%20Cookie%EF%BC%8C%E7%94%A8%E4%BA%8E%E6%97%85%E8%A1%8C%20skill%20%E5%A2%9E%E5%BC%BA%E6%A3%80%E7%B4%A2)
- [Set DOUYIN_COOKIE](minis://settings/environments?create_key=DOUYIN_COOKIE&create_value=&create_note=%E6%8A%96%E9%9F%B3%E7%BD%91%E9%A1%B5%E7%99%BB%E5%BD%95%20Cookie%EF%BC%8C%E7%94%A8%E4%BA%8E%E6%97%85%E8%A1%8C%20skill%20%E5%A2%9E%E5%BC%BA%E6%A3%80%E7%B4%A2)

## 平台使用建议

### 小红书
- 优先搜公开可见页面与搜索引擎索引页。
- 适合提取：出片点、避坑、美食、路线、预约提醒。
- 遇到登录墙/风控时，明确说明并降级。

**推荐解决路径：**
1. 默认路径：搜索引擎检索小红书公开索引页与分享页。
2. 增强路径：若用户浏览器内已登录小红书，可尝试复用站点 cookie 读取更多公开页面。
3. 降级路径：若搜索/正文受限，改用高德 + 大众点评 + 马蜂窝 + 官方页交叉补齐。
4. 提取策略：不追求全量抓取，只提炼高频景点、高频踩坑、高频预约提醒、高频店名。
5. 可信度策略：小红书更适合“经验层信息”，不要把单帖内容直接当官方事实。

### 抖音
- 适合提取：短视频热点、机位、近期热门玩法、夜景/美食打卡。
- 动态加载或限制较多时，不做稳定性承诺。

**推荐解决路径：**
1. 默认路径：优先搜索公开分享页、搜索引擎索引结果、可读视频介绍页。
2. 增强路径：若用户浏览器内已登录抖音，可尝试在登录态下读取公开可见内容。
3. 降级路径：若网页端无法稳定展开内容，则只把抖音作为“热度发现器”，再用地图/点评/官方页核实。
4. 提取策略：重点提取近期热门点、夜景玩法、拍照机位、店铺热词，不依赖逐字稿。
5. 可信度策略：抖音适合发现趋势，不适合作为营业时间、预约规则、票价的唯一来源。

### 高德地图
- 适合：POI、营业时间、评分、地理聚类、顺路安排。
- 国内目的地优先使用。

### Google Maps
- 适合：境外目的地、评论摘要、营业信息、地图关系。
- 国内数据不一定优于高德。

### 飞猪 / 携程 / Booking
- 适合：住宿区域、价格带、预订参考。
- 不要把实时价格当绝对值；表述为“参考区间”。

### 大众点评
- 适合：餐饮与本地店铺交叉验证。
- 注意区分“网红热度”和“稳定口碑”。

### 马蜂窝
- 适合：传统攻略、经典路线、目的地 overview。

### 其他建议平台
- Google Travel：适合境外目的地的大盘概览与路线草图。
- Tripadvisor：适合境外景点/餐饮评论交叉验证。
- YouTube：适合境外或公开视频型旅行攻略补充。
- Bilibili：适合中文长视频攻略与路线讲解。
- Rome2Rio：适合跨城交通方式比较。
- 官方页面：开放时间、预约、票务、闭馆公告优先以官方为准。
- 天气服务：决定夜景、户外、公园、博物馆替代方案。

## 执行要求

- 尽量给出“为什么这样排”的解释，而不是只列点。
- 避免把多个相距过远的点硬塞进同一天。
- 有儿童、老人、雨天、高温等约束时，主动调整节奏。
- 预算不明确时，可按“经济 / 舒适 / 宽裕”三档输出。
- 没有可靠来源支持的说法，不要伪装成事实。

## 工具建议

- 需要网页搜索时，优先使用 `web-search` skill。
- 需要 HTML 展示时，优先使用 `generative-ui-minis` skill。
- 需要读取页面正文、抓公开链接、做多页面汇总时，使用浏览器工具或 `minis-browser-use`。
- 如需长期复用模板，可把 JSON/HTML 输出写到 `/var/minis/workspace/`。
- 需要更细的平台判断时，按需读取：
  - `references/platforms.md`
  - `references/planning-strategies.md`

## 不做的事

- 不承诺绕过平台风控。
- 不把私有/需登录受限内容当作默认可访问数据源。
- 不把实时价格、余票、营业时间说成绝对准确，除非已明确核实。
- 不为了“像原项目”而引入不必要的复杂前后端结构。
